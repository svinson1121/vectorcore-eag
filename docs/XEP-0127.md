# VectorCore EAG — XEP-0127: CAP Over XMPP

**Implementation Reference for Client Developers**

---

## Overview

VectorCore EAG distributes emergency alerts in real time over XMPP using the
[XEP-0127](https://xmpp.org/extensions/xep-0127.html) extension. EAG operates
as the **XMPP server** — clients connect inbound, authenticate with credentials
configured in the EAG web UI, and receive a continuous push stream of CAP 1.2
alert stanzas. No polling or subscription management is required — once
connected and authenticated, alerts are delivered immediately as they are
ingested.

This document covers what clients need to implement to receive alerts from EAG.
For the full connection protocol (stream negotiation, SASL, TLS), see
[`XMPP.md`](XMPP.md).

---

## Connection

Clients connect to the EAG XMPP server using credentials configured in the
EAG web UI under **CAP XMPP Peers**. Each peer has a username and password.
The peer's XMPP JID is formed as `<username>@<domain>`.

EAG exposes up to two independent TCP listeners, configured in `config.yaml`:

### c2s — Plain TCP with optional STARTTLS (default port 5222)

```yaml
xmpp_server:
  domain: "eag.example.com"
  c2s:
    enabled: true
    port: 5222
    starttls: false   # true = STARTTLS offered and required (needs tls.cert + tls.key)
                      # false = plain TCP, no encryption (trusted networks only)
```

### c2s_tls — Direct TLS (default port 5223)

```yaml
xmpp_server:
  c2s_tls:
    enabled: true
    port: 5223        # TLS from the first byte — requires tls.cert + tls.key
```

### TLS Certificate

```yaml
xmpp_server:
  tls:
    cert: "/etc/vectorcore/eag.crt"
    key:  "/etc/vectorcore/eag.key"
```

TLS 1.2 minimum is enforced. Self-signed certificates are supported — clients
may need to disable certificate verification or explicitly trust the server's CA.

For the detailed step-by-step connection flow and XML protocol exchange, see
[`XMPP.md`](XMPP.md).

## Message Stanza Format

Once authenticated, EAG pushes CAP alerts as XMPP `<message>` stanzas of type
`headline`. The `to` attribute contains the peer's full JID
(`username@domain/resource`).

```xml
<message to="ops-client@eag.example.com/my-client" type="headline">
  <alert xmlns="urn:oasis:names:tc:emergency:cap:1.2">
    ...
  </alert>
</message>
```

**Key points:**

- `type="headline"` per XEP-0127 §4. Headline messages are broadcast-style;
  no acknowledgement is required and they are not stored in offline queues.
- The `<alert>` element is a **direct child** of `<message>` with the CAP 1.2
  namespace declared inline.
- No `<body>` element is present. Clients must parse the `<alert>` XML child.
- All text values are XML-entity-escaped (`&amp;` `&lt;` `&gt;` `&quot;` `&apos;`).
- All datetimes are RFC 3339 with timezone offset. Do not assume UTC.
- `Cancel` alerts are **not forwarded** — they are processed internally.
- Clients send no alert stanzas. Any non-negotiation stanzas sent by clients
  are silently discarded.

## CAP 1.2 Payload

The `<alert>` element conforms to CAP 1.2
(`urn:oasis:names:tc:emergency:cap:1.2`).

### Full Example

```xml
<message to="ops-client@eag.example.com/my-client" type="headline">
  <alert xmlns="urn:oasis:names:tc:emergency:cap:1.2">
    <identifier>urn:oid:2.49.0.1.840.0.abc123def456</identifier>
    <sender>w-nws.webmaster@noaa.gov</sender>
    <sent>2026-04-14T12:00:00-05:00</sent>
    <status>Actual</status>
    <msgType>Alert</msgType>
    <scope>Public</scope>
    <info>
      <language>en-US</language>
      <category>Met</category>
      <event>Tornado Warning</event>
      <responseType>Shelter</responseType>
      <urgency>Immediate</urgency>
      <severity>Extreme</severity>
      <certainty>Observed</certainty>
      <effective>2026-04-14T12:00:00-05:00</effective>
      <onset>2026-04-14T12:00:00-05:00</onset>
      <expires>2026-04-14T12:45:00-05:00</expires>
      <senderName>NWS Birmingham AL</senderName>
      <headline>Tornado Warning issued April 14 at 12:00PM CDT</headline>
      <description>A tornado has been observed...</description>
      <instruction>Take shelter immediately in an interior room...</instruction>
      <parameter>
        <valueName>affectedZones</valueName>
        <value>https://api.weather.gov/zones/county/ALJ001</value>
      </parameter>
      <area>
        <areaDesc>Jefferson; Shelby</areaDesc>
        <geocode>
          <valueName>SAME</valueName>
          <value>001073</value>
        </geocode>
        <geocode>
          <valueName>SAME</valueName>
          <value>001117</value>
        </geocode>
        <geocode>
          <valueName>UGC</valueName>
          <value>ALJ001</value>
        </geocode>
        <geocode>
          <valueName>UGC</valueName>
          <value>ALC117</value>
        </geocode>
      </area>
    </info>
  </alert>
</message>
```

### Alert-Level Fields

| Element | Always Present | Description |
|---|---|---|
| `identifier` | yes | Globally unique CAP ID — use as dedup key |
| `sender` | yes | Originating sender address |
| `sent` | yes | RFC 3339 — time the alert was originally issued |
| `status` | yes | `Actual` `Exercise` `System` `Test` `Draft` |
| `msgType` | yes | `Alert` or `Update` (`Cancel` is never forwarded) |
| `scope` | yes | `Public` `Restricted` `Private` |
| `references` | no | Space-separated prior alert IDs superseded by an Update |

### Info-Level Fields

| Element | Always Present | Description |
|---|---|---|
| `language` | yes | Always `en-US` |
| `category` | yes | `Geo` `Met` `Safety` `Security` `Rescue` `Fire` `Health` `Env` `Transport` `Infra` `CBRNE` `Other` |
| `event` | yes | Free-text event type, e.g. `Tornado Warning` |
| `urgency` | yes | `Immediate` `Expected` `Future` `Past` `Unknown` |
| `severity` | yes | `Extreme` `Severe` `Moderate` `Minor` `Unknown` |
| `certainty` | yes | `Observed` `Likely` `Possible` `Unlikely` `Unknown` |
| `effective` | no | RFC 3339 — when the alert becomes effective |
| `onset` | no | RFC 3339 — expected start of hazardous condition |
| `expires` | no | RFC 3339 — when the alert expires |
| `responseType` | no | `Shelter` `Evacuate` `Prepare` `Execute` `Avoid` `Monitor` `Assess` `AllClear` `None` |
| `senderName` | no | Human-readable name of the issuing organisation |
| `headline` | no | Short human-readable summary |
| `description` | no | Full plain-text description of the hazard |
| `instruction` | no | Recommended public action |
| `parameter` | no | Repeatable. Source-specific extensions — see below |
| `area/areaDesc` | no | Human-readable affected area description |
| `area/geocode` | no | Repeatable. `<valueName>` is `SAME` or `UGC`; `<value>` is the code |

> All datetimes are RFC 3339 / ISO 8601 with timezone offset. Parse with a
> standards-compliant datetime library; do not assume UTC.

### Parameters

`<parameter>` elements carry source-specific data that does not map to a
standard CAP field. Each `<parameter>` has a `<valueName>` and a `<value>`.
Clients should treat unrecognised parameter names as advisory and ignore them
gracefully.

| `valueName` | Source | Description |
|---|---|---|
| `affectedZones` | NWS | URL to an NWS zone resource. May appear multiple times. |

Additional parameter names may be present from other feed sources. Iterate all
`<parameter>` children and key them by `<valueName>` to future-proof your
implementation.

## Message Types

### Alert

A new alert. Insert or display as a new event.

### Update

Supersedes one or more prior alerts. The `<references>` element lists the IDs
of the alerts being replaced. Clients should:

1. Display or store the new alert.
2. Mark or remove the referenced prior alerts as superseded.

```xml
<references>w-nws.webmaster@noaa.gov,urn:oid:2.49.0.1.840.0.prev1,2026-04-13T17:30:00-05:00</references>
```

References are space-separated tuples of `sender,identifier,sent`.

### Cancel

Cancels one or more prior alerts identified in `<references>`.

> **EAG does not forward Cancel messages over XMPP.** Cancel processing is
> handled internally — referenced alerts are soft-deleted in the EAG database.
> Clients connected directly to upstream XMPP feeds (bypassing EAG) may receive
> Cancel stanzas and should handle them as described above.

## Severity Values

Severity is used by EAG for per-peer filtering and should be used by clients
for visual prioritisation.

| Value | Meaning | Suggested Colour |
|---|---|---|
| `Extreme` | Extraordinary threat to life or property | Red `#ff4444` |
| `Severe` | Significant threat to life or property | Orange `#ff7c2a` |
| `Moderate` | Possible threat to life or property | Yellow `#f5d623` |
| `Minor` | Minimal threat | Blue `#3a8fd4` |
| `Unknown` | Severity not known | Grey `#5a6a7e` |

## Per-Peer Filtering

Each peer account in EAG has optional server-side filters configured in the
EAG web UI under **CAP XMPP Peers**. Alerts that do not match **all** active
filters for a peer are not forwarded to that peer. Clients receive only
pre-filtered alerts — no client-side re-filtering is needed.

| Filter | Match Type | CAP Field |
|---|---|---|
| Severity | Exact membership | `<severity>` |
| Event | Substring | `<event>` |
| Area | Substring | `<areaDesc>` |
| Status | Exact membership | `<status>` |

An empty filter list means "match all". Filters are combined with AND logic —
all non-empty filters must pass for an alert to be delivered.

## Receiving Alerts — Client Implementation

### 1. Connect and Authenticate

Connect to the EAG XMPP server on the configured host and port. Authenticate
using the username and password configured for your peer account in the EAG
web UI. See [`XMPP.md`](XMPP.md) for the complete stream negotiation flow.

### 2. Listen for Headline Messages

Listen for incoming `<message>` stanzas. Filter for:

```
stanza.type == "headline"
```

### 3. Extract the CAP Payload

Check for a child element with local name `alert` and namespace
`urn:oasis:names:tc:emergency:cap:1.2`:

```python
# Python / lxml
NS = "urn:oasis:names:tc:emergency:cap:1.2"
alert_el = message_stanza.find(f"{{{NS}}}alert")
if alert_el is not None:
    process_cap(alert_el)
```

```javascript
// JavaScript / DOM
const NS = "urn:oasis:names:tc:emergency:cap:1.2"
const alertEl = stanza.getElementsByTagNameNS(NS, "alert")[0]
if (alertEl) processCAP(alertEl)
```

```go
// Go / encoding/xml
type CAPAlert struct {
    XMLName    xml.Name  `xml:"urn:oasis:names:tc:emergency:cap:1.2 alert"`
    Identifier string    `xml:"identifier"`
    Sender     string    `xml:"sender"`
    Sent       string    `xml:"sent"`
    Status     string    `xml:"status"`
    MsgType    string    `xml:"msgType"`
    Scope      string    `xml:"scope"`
    References string    `xml:"references"`
    Info       []CAPInfo `xml:"info"`
}

type CAPInfo struct {
    Language     string       `xml:"language"`
    Category     string       `xml:"category"`
    Event        string       `xml:"event"`
    ResponseType string       `xml:"responseType"`
    Urgency      string       `xml:"urgency"`
    Severity     string       `xml:"severity"`
    Certainty    string       `xml:"certainty"`
    Effective    string       `xml:"effective"`
    Onset        string       `xml:"onset"`
    Expires      string       `xml:"expires"`
    SenderName   string       `xml:"senderName"`
    Headline     string       `xml:"headline"`
    Description  string       `xml:"description"`
    Instruction  string       `xml:"instruction"`
    Parameters   []CAPParam   `xml:"parameter"`
    Areas        []CAPArea    `xml:"area"`
}

type CAPParam struct {
    ValueName string `xml:"valueName"`
    Value     string `xml:"value"`
}

type CAPArea struct {
    AreaDesc string     `xml:"areaDesc"`
    Geocodes []CAPParam `xml:"geocode"` // same shape as parameter
}

// Inside your message handler:
type IncomingMessage struct {
    Alert *CAPAlert `xml:"urn:oasis:names:tc:emergency:cap:1.2 alert"`
}
var m IncomingMessage
dec.DecodeElement(&m, &startElement)
if m.Alert != nil && len(m.Alert.Info) > 0 {
    info := m.Alert.Info[0]
    // Access standard fields: info.Event, info.Severity, etc.

    // Access geocodes
    for _, gc := range info.Areas[0].Geocodes {
        fmt.Printf("geocode %s = %s\n", gc.ValueName, gc.Value)
    }

    // Access parameters (source-specific extensions)
    for _, p := range info.Parameters {
        fmt.Printf("param %s = %s\n", p.ValueName, p.Value)
    }
}
```

### 4. Deduplication

Use `<identifier>` as the unique key. EAG will not send duplicate alerts for
the same identifier in a single session, but clients operating across reconnects
should guard against processing the same identifier twice by maintaining a seen
set or checking against stored records.

### 5. Handle Updates

When `msgType` is `Update`, parse the `<references>` element and mark or remove
the referenced prior alerts as superseded before inserting the new alert.

### 6. Handle Expiry

The `<expires>` field is an absolute datetime. Clients should hide or visually
demote alerts past their expiry. EAG performs server-side expiry sweeps and
will not re-forward expired alerts, but alerts already displayed to users may
expire while on-screen.

---

## Reconnection and Unforwarded Alert Recovery

EAG tracks whether each alert has been forwarded to connected peers. When a
client reconnects and re-authenticates, EAG immediately sweeps all unforwarded
alerts for that peer and re-delivers them. This covers short outages without
requiring any special client behaviour — simply reconnect.

**Client reconnection guidance:**

- On TCP disconnect, delay briefly (1–5 seconds) then reconnect and
  re-authenticate from the beginning of the stream negotiation flow.
- Use exponential backoff for repeated failures (e.g. 2s, 4s, 8s … up to 60s).
- After reconnecting, be prepared to receive a burst of queued alerts.

**Limitation:** Alerts that have already expired by the time the client
reconnects will not be re-delivered (EAG's expiry worker soft-deletes them).

---

## Concurrency

One active session is permitted per peer username. If a second client connects
with the same credentials, the existing session is terminated and replaced.
All pending (unforwarded) alerts are then delivered to the new session.

## Client Code Examples

### Python — `slixmpp`

```python
import slixmpp
import asyncio

CAP_NS = "urn:oasis:names:tc:emergency:cap:1.2"

class CAPClient(slixmpp.ClientXMPP):
    def __init__(self, jid, password, server, port, use_tls=False):
        super().__init__(jid, password)
        self._server = server
        self._port = port
        self._use_tls = use_tls
        self.add_event_handler("session_start", self.on_start)
        self.add_event_handler("message", self.on_message)

    async def on_start(self, event):
        # Receive-only client — no presence needed
        pass

    def on_message(self, msg):
        if msg["type"] != "headline":
            return
        alert = msg.xml.find(f"{{{CAP_NS}}}alert")
        if alert is None:
            return
        identifier = alert.findtext(f"{{{CAP_NS}}}identifier")
        info       = alert.find(f"{{{CAP_NS}}}info")
        if info is None:
            return
        event_type    = info.findtext(f"{{{CAP_NS}}}event")
        severity      = info.findtext(f"{{{CAP_NS}}}severity")
        urgency       = info.findtext(f"{{{CAP_NS}}}urgency")
        expires       = info.findtext(f"{{{CAP_NS}}}expires")
        headline      = info.findtext(f"{{{CAP_NS}}}headline")
        instruction   = info.findtext(f"{{{CAP_NS}}}instruction")
        sender_name   = info.findtext(f"{{{CAP_NS}}}senderName")
        area          = info.find(f"{{{CAP_NS}}}area")
        area_desc     = area.findtext(f"{{{CAP_NS}}}areaDesc") if area is not None else None

        # Geocodes (SAME / UGC codes)
        geocodes = {}
        if area is not None:
            for gc in area.findall(f"{{{CAP_NS}}}geocode"):
                name = gc.findtext(f"{{{CAP_NS}}}valueName")
                val  = gc.findtext(f"{{{CAP_NS}}}value")
                geocodes.setdefault(name, []).append(val)

        # Source-specific parameters
        params = {}
        for p in info.findall(f"{{{CAP_NS}}}parameter"):
            name = p.findtext(f"{{{CAP_NS}}}valueName")
            val  = p.findtext(f"{{{CAP_NS}}}value")
            params.setdefault(name, []).append(val)

        print(f"[{severity}/{urgency}] {event_type} — expires {expires}")
        print(f"  Area:     {area_desc}")
        print(f"  Sender:   {sender_name}")
        print(f"  Headline: {headline}")
        if instruction:
            print(f"  Action:   {instruction}")
        if "SAME" in geocodes:
            print(f"  SAME:     {', '.join(geocodes['SAME'])}")
        print(f"  ID:       {identifier}")

async def main():
    # Plain TCP (starttls: false)
    client = CAPClient(
        jid="ops-client@eag.example.com",
        password="s3cr3t",
        server="eag.example.com",
        port=5222,
    )
    client.connect((client._server, client._port), use_ssl=False)
    await client.disconnected

asyncio.run(main())
```

For STARTTLS (`starttls: true`), pass `use_ssl=False` and ensure slixmpp
negotiates STARTTLS automatically (it will if the server advertises it).

For direct TLS (`c2s_tls`), connect on port 5223 with `use_ssl=True`.

### Go — stdlib `net` + `encoding/xml`

```go
package main

import (
    "encoding/base64"
    "encoding/xml"
    "fmt"
    "net"
)

const (
    server   = "eag.example.com:5222"
    username = "ops-client"
    password = "s3cr3t"
    domain   = "eag.example.com"
    resource = "go-client"

    nsStream = "http://etherx.jabber.org/streams"
    nsClient = "jabber:client"
    nsSASL   = "urn:ietf:params:xml:ns:xmpp-sasl"
    nsBind   = "urn:ietf:params:xml:ns:xmpp-bind"
    nsCAP    = "urn:oasis:names:tc:emergency:cap:1.2"
)

type CAPAlert struct {
    XMLName    xml.Name `xml:"alert"`
    Identifier string   `xml:"identifier"`
    Sender     string   `xml:"sender"`
    Sent       string   `xml:"sent"`
    Status     string   `xml:"status"`
    MsgType    string   `xml:"msgType"`
    References string   `xml:"references"`
    Info       CAPInfo  `xml:"info"`
}

type CAPInfo struct {
    Event       string `xml:"event"`
    Urgency     string `xml:"urgency"`
    Severity    string `xml:"severity"`
    Certainty   string `xml:"certainty"`
    Expires     string `xml:"expires"`
    Headline    string `xml:"headline"`
    Description string `xml:"description"`
    Area        struct {
        AreaDesc string `xml:"areaDesc"`
    } `xml:"area"`
}

func saslPlain(username, password string) string {
    raw := "\x00" + username + "\x00" + password
    return base64.StdEncoding.EncodeToString([]byte(raw))
}

func main() {
    conn, err := net.Dial("tcp", server)
    if err != nil {
        panic(err)
    }
    defer conn.Close()

    send := func(s string) { fmt.Fprint(conn, s) }

    streamOpen := func() {
        fmt.Fprintf(conn,
            "<?xml version='1.0'?><stream:stream to=%q "+
                "xmlns=%q xmlns:stream=%q version='1.0'>",
            domain, nsClient, nsStream)
    }

    skipToStart := func(dec *xml.Decoder) xml.StartElement {
        for {
            tok, _ := dec.Token()
            if s, ok := tok.(xml.StartElement); ok {
                return s
            }
        }
    }

    // ── SASL authentication ───────────────────────────────────────────────────
    streamOpen()
    dec := xml.NewDecoder(conn)

    skipToStart(dec) // <stream:stream>
    skipToStart(dec) // <stream:features>
    dec.Skip()       //nolint:errcheck

    send(fmt.Sprintf(`<auth xmlns=%q mechanism="PLAIN">%s</auth>`,
        nsSASL, saslPlain(username, password)))

    result := skipToStart(dec) // <success/> or <failure/>
    if result.Name.Local != "success" {
        fmt.Println("authentication failed")
        return
    }

    // ── Resource binding ──────────────────────────────────────────────────────
    dec = xml.NewDecoder(conn)
    streamOpen()

    skipToStart(dec) // <stream:stream>
    skipToStart(dec) // <stream:features>
    dec.Skip()       //nolint:errcheck

    send(fmt.Sprintf(
        `<iq type='set' id='bind_1'><bind xmlns=%q><resource>%s</resource></bind></iq>`,
        nsBind, resource))

    skipToStart(dec) // <iq type='result'>
    dec.Skip()       //nolint:errcheck

    fmt.Println("connected — waiting for alerts...")

    // ── Receive alerts ────────────────────────────────────────────────────────
    for {
        tok, err := dec.Token()
        if err != nil {
            fmt.Println("disconnected:", err)
            return
        }
        start, ok := tok.(xml.StartElement)
        if !ok {
            continue
        }
        if start.Name.Local != "message" {
            dec.Skip() //nolint:errcheck
            continue
        }
        var m struct {
            Alert *CAPAlert `xml:"urn:oasis:names:tc:emergency:cap:1.2 alert"`
        }
        dec.DecodeElement(&m, &start) //nolint:errcheck
        if m.Alert == nil {
            continue
        }
        a := m.Alert
        fmt.Printf("[%s] %s — %s/%s\n  Area:    %s\n  Expires: %s\n  ID:      %s\n\n",
            a.Info.Severity, a.Info.Event, a.Info.Urgency, a.Info.Certainty,
            a.Info.Area.AreaDesc, a.Info.Expires, a.Identifier)
    }
}
```

### JavaScript — Node.js `@xmpp/client`

```javascript
const { client, xml } = require('@xmpp/client')

const CAP_NS = 'urn:oasis:names:tc:emergency:cap:1.2'

// Plain TCP — service is 'xmpp://' for unencrypted or STARTTLS
// Use 'xmpps://' for direct TLS on port 5223
const xmpp = client({
  service:  'xmpp://eag.example.com:5222',
  domain:   'eag.example.com',
  username: 'ops-client',
  password: 's3cr3t',
  resource: 'js-client',
})

xmpp.on('online', () => {
  console.log('connected — waiting for alerts...')
})

xmpp.on('stanza', stanza => {
  if (!stanza.is('message') || stanza.attrs.type !== 'headline') return

  const alertEl = stanza.getChild('alert', CAP_NS)
  if (!alertEl) return

  const identifier  = alertEl.getChildText('identifier')
  const info        = alertEl.getChild('info')
  if (!info) return

  const event       = info.getChildText('event')
  const severity    = info.getChildText('severity')
  const urgency     = info.getChildText('urgency')
  const expires     = info.getChildText('expires')
  const headline    = info.getChildText('headline')
  const instruction = info.getChildText('instruction')
  const senderName  = info.getChildText('senderName')
  const area        = info.getChild('area')
  const areaDesc    = area?.getChildText('areaDesc')

  // Geocodes (SAME / UGC codes)
  const geocodes = {}
  area?.getChildren('geocode').forEach(gc => {
    const name = gc.getChildText('valueName')
    const val  = gc.getChildText('value')
    if (name) (geocodes[name] = geocodes[name] || []).push(val)
  })

  // Source-specific parameters
  const params = {}
  info.getChildren('parameter').forEach(p => {
    const name = p.getChildText('valueName')
    const val  = p.getChildText('value')
    if (name) (params[name] = params[name] || []).push(val)
  })

  console.log(`[${severity}/${urgency}] ${event}`)
  console.log(`  Area:     ${areaDesc}`)
  console.log(`  Sender:   ${senderName}`)
  console.log(`  Headline: ${headline}`)
  if (instruction) console.log(`  Action:   ${instruction}`)
  if (geocodes.SAME) console.log(`  SAME:     ${geocodes.SAME.join(', ')}`)
  console.log(`  Expires:  ${expires}`)
  console.log(`  ID:       ${identifier}`)
})

xmpp.on('error', err => {
  console.error('xmpp error:', err)
})

xmpp.start().catch(console.error)
```

## Notes and Limitations

- **PLAIN only.** No SCRAM, DIGEST-MD5, or GSSAPI. Use STARTTLS or direct TLS
  to protect credentials in transit on untrusted networks.
- **No offline delivery.** Alerts published while a client is disconnected are
  not queued at the XMPP layer. EAG re-delivers unforwarded alerts on reconnect,
  but alerts that expired during the outage are not re-sent.
- **Receive-only.** Clients send no alert stanzas. Incoming stanzas from clients
  (other than stream negotiation IQs) are silently discarded.
- **One session per username.** A new login with the same credentials terminates
  the existing session. All pending alerts are delivered to the new session.
- **No roster, presence, or MUC.** EAG implements only the RFC 6120 subset
  needed for authenticated, receive-only CAP delivery.
- **No polygon in XMPP payload.** GeoJSON geometry is stored internally but not
  included in the XMPP stanza. Clients requiring polygon data should query the
  EAG REST API: `GET /api/v1/alerts/{id}`.
- **Cancel not forwarded.** `msgType=Cancel` stanzas are never sent over XMPP.
  Cancel processing (soft-deleting referenced alerts) is handled internally.
